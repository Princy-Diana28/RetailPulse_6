# 📊 RetailPulse — AI-Powered Demand & Inventory Intelligence Platform

An end-to-end retail analytics platform built on the **Online Retail II** dataset:
data cleaning, feature engineering, EDA, customer segmentation, demand
forecasting, churn analysis, inventory optimization, and a full interactive
Streamlit dashboard — all backed by a modular, testable Python codebase.

---

## Project Overview

RetailPulse turns ~1.07 million raw e-commerce transaction lines from a
UK-based online gift retailer (Dec 2009 – Dec 2011, 43 countries) into a
decision-ready analytics product. It is built the way a small data team would
actually ship this internally: a reusable pipeline in `src/`, a cached
Streamlit front end in `app.py` / `views/`, and an EDA notebook that calls the
exact same modules the app uses — so notebook, dashboard, and any future
batch job never disagree with each other.

Every number in this README below the dataset section was generated by
running the actual pipeline against the actual bundled dataset — nothing is
hypothetical.

## Features

- **Data Cleaning** — deduplication, missing-`CustomerID` handling,
  cancellation-invoice flagging, invalid quantity/price filtering, non-product
  stock-code removal (postage, bank charges, etc.), IQR-based outlier
  winsorization.
- **Feature Engineering** — calendar features (month, quarter, day-of-week,
  season), `DemandLevel` buckets, order value, customer lifetime value,
  purchase frequency, recency.
- **Exploratory Data Analysis** — monthly revenue trend, top products (by
  revenue and volume), sales by country, seasonal distribution, revenue and
  quantity histograms, correlation heatmap.
- **Customer Segmentation** — RFM (Recency/Frequency/Monetary) features
  clustered with KMeans, automatically labeled by relative segment value
  (Champions, Loyal Customers, At Risk, etc.) — labels are re-derived from
  cluster statistics every run, never hardcoded to a cluster index.
- **Demand Forecasting** — Holt-Winters (statsmodels) is the default forecasting
  engine, chosen specifically for its light memory footprint on free-tier
  deployments. **Prophet is available as an opt-in toggle** on the Demand
  Forecasting page for richer seasonality modeling when memory headroom allows.
- **Churn Analysis** — an inactivity-based analytical label (configurable
  threshold) plus a Random Forest classifier trained on a **time-based,
  leakage-free holdout** (features computed before a snapshot date, label
  determined by behavior strictly after it).
- **Inventory Optimization** — per-SKU stock action recommendations
  (Increase / Maintain / Reduce Stock) comparing trailing vs. prior 30-day
  demand, with a safety-stock reorder quantity using a standard
  service-level formula.
- **Automated Business Insights** — plain-English findings generated
  directly from the data (top/bottom revenue months, top market, fastest
  growing/declining products, most valuable segment, inventory actions
  needed).
- **Model Performance Dashboard** — silhouette score, classifier
  accuracy/precision/recall/F1/ROC-AUC, confusion matrix, feature
  importance, forecast fit error (MAE/MAPE).
- **Downloads everywhere** — clean dataset, feature-engineered dataset,
  segmentation results, forecast results, churn results, inventory
  recommendations — all exportable as CSV from the relevant page.
- **Premium dashboard UI** — grouped sidebar navigation (Overview, Analytics,
  AI Models, Reporting, Platform) with clean Material icons, a consistent
  dark theme, KPI cards with hover states, and uniform page headers across
  every section.

## Project Architecture

RetailPulse follows a layered architecture, so each concern stays isolated
and testable on its own:

```
┌─────────────────────────────────────────────────────────────────────┐
│  Data Layer            data/raw → src/preprocessing.py → data/processed│
├─────────────────────────────────────────────────────────────────────┤
│  Feature Layer         src/feature_engineering.py (RFM, CLV, calendar)│
├─────────────────────────────────────────────────────────────────────┤
│  Model Layer           segmentation.py · forecasting.py · churn.py ·  │
│                        inventory.py → saved_models/                   │
├─────────────────────────────────────────────────────────────────────┤
│  Orchestration Layer   src/pipeline.py (cached, Streamlit-aware)       │
├─────────────────────────────────────────────────────────────────────┤
│  Presentation Layer    app.py (router) → views/*.py (11 pages)        │
└─────────────────────────────────────────────────────────────────────┘
```

- **Data Layer** — raw Online Retail II transactions are cleaned once
  (dedup, missing-value handling, cancellation/return flagging, outlier
  winsorization) and cached as Parquet for fast reloads.
- **Feature Layer** — builds the RFM, calendar, and lifetime-value features
  every downstream model consumes, so all models see identical inputs.
- **Model Layer** — four independent, swappable models: KMeans segmentation,
  Prophet/Holt-Winters forecasting, Random Forest churn classification, and
  a rules-based inventory optimizer.
- **Orchestration Layer** — `src/pipeline.py` wraps every stage in
  `st.cache_data`, so switching between pages never re-runs cleaning,
  clustering, or model training unless the underlying data changes.
- **Presentation Layer** — `app.py` is a thin router (`st.navigation`) over
  eleven independent page modules in `views/`, each responsible for exactly
  one part of the dashboard.

## Installation

Requires **Python 3.10+**.

```bash
git clone <your-repo-url> RetailPulse
cd RetailPulse
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

## How to Run

```bash
streamlit run app.py
```

The app opens at `http://localhost:8501` with the bundled Online Retail II
dataset pre-loaded — every page works immediately, no upload required. Use
the **Dataset Upload** page to swap in your own retail transaction export
(CSV/XLSX with the same column schema) and the entire pipeline re-runs
against it.

## Dataset

**Online Retail II** — a real transactional dataset from a UK-based,
registered non-store online retailer, covering **01/12/2009 – 09/12/2011**.
Columns: `Invoice, StockCode, Description, Quantity, InvoiceDate, Price,
Customer ID, Country`.

Bundled at `data/raw/online_retail_II.xlsx` (two sheets, one per year) with a
cached Parquet snapshot (`online_retail_ii_combined.parquet`) for fast app
cold starts. On first run without a cache, the app parses the raw Excel file
and writes the cache automatically.

**Data caveats worth knowing:**
- The dataset ends mid-December 2011 — that final month is partial, so
  month-over-month comparisons involving it should be read accordingly.
- ~23% of raw rows have no `Customer ID` and are excluded from
  customer-level analysis (segmentation, churn, CLV) since they can't be
  attributed to a person — they remain in product/sales-level analytics
  wherever `CustomerID` isn't required.
- Invoices prefixed `C` are cancellations and are excluded from the core
  sales table (flagged separately in the cleaning report).

## Future Improvements

- Swap the top-N heuristic inventory model for a per-SKU Prophet/ARIMA
  forecast where transaction volume supports it.
- Add a model registry (e.g. MLflow) for versioned experiment tracking, as
  sketched in the original architecture notes.
- Expose a FastAPI serving layer alongside the Streamlit UI for programmatic
  access to forecasts and recommendations.
- Add drift detection on incoming uploads (schema + distribution checks)
  before running the pipeline.
- Extend churn modeling with survival analysis (time-to-churn) instead of a
  fixed inactivity threshold.

## Deployment Instructions

**Streamlit Community Cloud**
1. Push this repository to GitHub.
2. On [share.streamlit.io](https://share.streamlit.io), create a new app
   pointing to `app.py` on your default branch.
3. No secrets are required for the default bundled dataset.

**Render**
1. Create a new Web Service from this repo.
2. Build command: `pip install -r requirements.txt`
3. Start command: `streamlit run app.py --server.port $PORT --server.address 0.0.0.0`

**Note on Prophet:** Holt-Winters (statsmodels) is the default forecasting
engine everywhere in the app, including the Home page KPI. This is a
deliberate memory choice, not a fallback — profiling showed Prophet's
compiled `cmdstan` subprocess adds roughly 400MB of peak memory on its own,
which is enough by itself to exceed free-tier limits on Streamlit Community
Cloud and Render. Prophet is still fully wired in and available as an opt-in
toggle on the Demand Forecasting page for anyone who wants it and has the
memory headroom (e.g. a paid tier or local run).
